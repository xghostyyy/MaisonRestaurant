# Claude Code: как провести одну сессию с максимальной отдачей

Стек и логика зафиксированы в `SPEC.md`. Этот файл — про **процесс** и готовые **промпты**.
Модель: **Sonnet 4.6**. Подход: spec-driven + plan mode + проектные skills + работа фазами
с коммитами и пушами.

---

## A. Подготовка (3 минуты до старта)

1. Создай пустую папку, положи в неё `SPEC.md` и этот файл, выполни `git init`,
   `git branch -M main`, `git remote add origin <URL>`. Залогинься в git (`gh auth login`
   или SSH-ключ) — это важно, чтобы Claude мог пушить без интерактивного ввода.
2. Убедись, что установлен Node.js LTS.
3. Запусти `claude` в этой папке.
4. Переключись в **Plan Mode** (`Shift+Tab`, пока не появится надпись Plan Mode).
   В этом режиме модель исследует и планирует, ничего не пишет — ты согласуешь план.
5. Полезные команды на сессию: `/skills`, `/review`, `/simplify`, `/compact`, `/clear`, `/doctor`.

**Почему это выжимает максимум:**
- **Plan mode** не даёт сразу писать код «как понял» — план фиксируется до строчки кода.
- **CLAUDE.md** (создаётся в Фазе 0) автоматически грузится в каждый запуск — конвенции
  не теряются между `/clear`.
- **Проектные skills** в `.claude/skills/` грузятся **по требованию**: повторяющиеся
  правила (дизайн-токены, алгоритм слотов, паттерны сервисов, безопасность форм)
  применяются единообразно и не съедают контекст, пока не нужны.
- **Коммиты и пуш после каждой фазы** = git как внешняя память. Между фазами можно
  `/compact` или `/clear` — состояние восстановится из git + CLAUDE.md + SPEC.md + skills.
- **Без TypeScript и без бандлеров** — стек выбран намеренно простым, чтобы модель
  меньше отвлекалась на конфигурацию и больше работала над предметной областью.

### Почему именно так используются «скиллы»

В Claude Code skill — это папка `.claude/skills/<имя>/SKILL.md` с YAML-frontmatter
(`name`, `description`) и инструкциями. Они грузятся **по требованию** (экономят
контекст), лежат в репозитории (одинаковы для всей команды) и дают **единообразие**
на повторяющихся задачах. Мы создаём пять проектных скиллов под самые рискованные
на консистентность места — дизайн «вкусно», ядро доступности столов, паттерны
SQLite-сервисов, безопасность форм, SSE — и активируем нужный в начале каждой
фазы. Плюс встроенные `/review`, `/simplify`, `/skills`. Это и есть «максимально
использовать скиллы»: не один длинный промпт, а переиспользуемые модули + spec
+ git, которые держат качество на всей сессии.
